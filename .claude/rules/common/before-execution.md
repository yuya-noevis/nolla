# Before Execution Rule (CRITICAL — 全やり取り・全タスクに適用)

## Yuyaの発言への対応（最重要）

Yuyaの指示・発言・提案を受けたら、即座に実行するな。まず以下を行う：

1. **事実/エビデンスに基づいているか検証する** → 根拠が弱ければ指摘する
2. **技術的に実現可能か確認する** → できないなら即座に伝える
3. **より良い方法がないか考える** → あるなら代替案を提示する
4. **問題やリスクがあれば指摘する** → 黙って従わない
5. **その上で提案する** → 中立的な判断を示す

「わかりました、やります」で着手するのは禁止。安易な同意禁止。言いなり禁止。

## タスク着手前の技術チェック

1. **このツールでこの品質は出せるか？** → 出せないなら即座に伝えて代替策を提示
2. **同じアプローチで過去に失敗していないか？** → 失敗済みなら別のアプローチを提案
3. **より効率的な方法はないか？** → あるなら先に提案
4. **パーツ分割・共通素材化で品質を担保できないか？** → できるなら提案

## 失敗時のルール
- 1回失敗 → 原因分析
- 2回失敗 → アプローチ変更（3回目の再試行禁止）
- 「もう1回やってみます」は禁止

## 開発作業の必須サイクル（CRITICAL — 例外なし）

コードを書いたら、必ず以下のサイクルを完了してからユーザーに確認を求める。
速度を理由にスキップすることは絶対に禁止。

### サイクル

1. **/tdd** — テストファーストで実装（新機能・バグ修正・リファクタリング時）
2. **/code-review** — コード品質・セキュリティレビュー
3. **/verify** — 6段階品質ゲート（Build/Type/Lint/Test/Security/Diff）
4. **実機相当チェック** — 下記のUI検証を全て通す
5. Git操作（commit、push）

### 実機相当チェック（デプロイ前に必ず実行）

UI変更を含む場合、以下を全て確認してからユーザーに見せる:

- [ ] 全画面のルーティングが正しく動作する（リンク切れなし）
- [ ] スマホ横向き（812x375）で全画面が見切れない
- [ ] iPad横向き（1194x834）で全画面が正しく表示される
- [ ] 全ボタン・タップターゲットが画面内に収まり操作可能
- [ ] ゲームフロー通し確認: 開始→プレイ→正解→次の問題→ラウンド完了→セッション完了
- [ ] 正解/不正解のフィードバックが表示される
- [ ] エラーレス学習の3段階が動作する
- [ ] チュートリアルが初回プレイ時に表示される
- [ ] 画面遷移が全て正しく動作する（戻る、ホームへ等）

### プッシュ前の必須Playwright検証（CRITICAL — スキップ厳禁）

git pushする前に、必ずPlaywrightで変更した機能の通しフローを実行する。
「ビルド成功」「テスト全パス」はプッシュの必要条件であって十分条件ではない。

1. devサーバーを起動する
2. Playwrightで変更箇所の画面を開く
3. 実際にクリック・入力して期待通り動くことを目視確認する
4. スクリーンショットを撮って証拠を残す
5. 確認できてからプッシュする

「コードを書いた → ビルド通った → プッシュ」は禁止。
「コードを書いた → ビルド通った → Playwrightで動作確認 → プッシュ」が正しい。

### 絶対禁止

- 品質ゲート（/verify）を通さずにコミットすること
- 実機チェックをせずにデプロイすること
- 「動くはず」でユーザーに確認を求めること — 動くことを自分で確認してから報告しろ
- ユーザーにバグを見つけさせること — それは開発リードの失格
- Playwright検証をせずにgit pushすること
