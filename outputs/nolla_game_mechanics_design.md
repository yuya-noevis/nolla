## 1. 共通設計原則（4ゲーム共通）

### 1.1 操作方式

| 項目 | 仕様 |
|------|------|
| 入力方式 | **タップのみ**。スワイプ・ドラッグ・ダブルタップ禁止 |
| タッチターゲット | 3-8歳: 最小 **64x64px**、9歳以上: 最小 48x48px |
| アイコン間余白 | 最低 **16px** |
| 画面の向き | **横向き（ランドスケープ）固定** |
| 1画面の情報要素 | **5個以下** |

### 1.2 フィードバック（200ms以内）

| 種類 | 正解時 | 不正解時 |
|------|--------|----------|
| 視覚 | 緑の光輪 + 要素が少し拡大（scale 1.1, 300ms） | **なし**（サイレント修正） |
| 音声 | 柔らかいチャイム音（ON/OFF切替可） | **なし** |
| 触覚 | 短い振動 15ms（ON/OFF切替可） | **なし** |
| 追加演出 | 星のパーティクル 3-5個（500ms で消滅） | 要素が元の位置に静かに戻る |

**禁止**: エラー音、減点、「間違い！」表示、画面の赤フラッシュ、スコア減少

### 1.3 コンテンツのランダム生成

**全ゲーム・全難度で、コンテンツは毎回ランダム生成する。**

- 同じ難度パラメータでも、カードの絵柄・配置・分類アイテム・シーン構成は毎回異なる
- 子どもが「答えを覚える」ことを防ぎ、常に認知訓練効果を維持する
- ランダム生成は難度パラメータの制約内で行う（例: 「ペア数4、視覚類似度20%」の制約でカード内容をランダム化）

### 1.4 エラーレス学習の実装パターン

全ゲーム共通で以下の3段階フォールバック:

```
不正解発生
  ↓
Stage 1: 要素が元の位置に静かに戻る（300ms）
  ↓ 同じ試行で2回目の不正解
Stage 2: 正解の選択肢が柔らかく脈動（pulse 1回、500ms）
  ↓ 同じ試行で3回目の不正解
Stage 3: 正解の選択肢に光の輪が表示され、ナビキャラが指さしアニメーション
  → 子どもがタップ → 通常の正解フィードバックを返す
```

**重要**: Stage 3 でも「間違えた」という印象を与えない。あくまで「ヒントが増える」演出。

### 1.5 チュートリアル（文字なし・4段階プロンプティング削減法）

各ゲーム初回プレイ時のみ。2回目以降はスキップ。

```
Step 1 [フルガイド]
  対象要素が発光（1秒）→ 周辺が暗くなる（視線誘導）
  → ナビキャラのハンドアニメーション（タップを示す）
  → 子どもがタップ → 正解フィードバック

Step 2 [ガイド軽減]
  対象要素が発光（0.5秒）→ ハンドアニメーションなし
  → 子どもがタップ → 正解フィードバック

Step 3 [最小ガイド]
  対象要素が一瞬だけ発光（0.3秒）
  → 子どもがタップ → 正解フィードバック

Step 4 [ガイドなし]
  通常のゲームプレイ。ガイドなし。
  → 正解 → チュートリアル完了、通常セッションへ
  → 不正解 → エラーレス学習 Stage 1-3 が発動
```

### 1.6 ゲーム画面の共通レイアウト

```
横向き 1194x834px（iPad基準）
┌──────────────────────────────────────────────────┐
│ [←戻る]                              [★ スター] │  ← ヘッダー（常時表示、高さ60px）
│                                                  │
│                                                  │
│              ゲーム固有エリア                      │  ← メインエリア（714px）
│              （各ゲームで定義）                    │
│                                                  │
│                                                  │
│                              [ナビキャラ]         │  ← フッター（60px）
└──────────────────────────────────────────────────┘

要素数: 3（ヘッダー / ゲームエリア / ナビキャラ）
ナビキャラは右下に小さく常駐。ヒント時のみ大きくなる。
```

## 2. 難度システム設計

### 2.1 設計思想

**離散的な「レベル」ではなく、連続的なパラメータ調整で難度を制御する。**

- 子どもには難度の概念を一切見せない
- 子どもはレベル選択ができない（ASD特有の同一性保持による簡単レベル固執を防止）
- 保護者・教師には具体的な到達度（パラメータ値ベース）を提示
- 内部では「IQバンド」を参照点として使用（開発・テスト・校正用）

**難度の粒度**:
- 各ゲームは3〜4個の独立パラメータを持ち、それぞれが連続的に変化する
- パラメータの組み合わせにより、各ゲームで**数万通り**の難度設定が存在する
- NCIのθ（0〜999.999）が子どもの現在地を**実質1000段階以上**の粒度で表現する
- 13バンドはあくまで開発・校正用の参照点。実際の難度調整はバンド間を滑らかに移動する
- **レベルMax = 定型発達の子供でも楽しめる難易度**。想定ユーザーのほとんどはそこに到達しない

### 2.2 各ゲームの難度パラメータ

各ゲームは複数の独立した難度パラメータ（ツマミ）を持つ。適応型システムが各パラメータを独立に調整する。

**神経衰弱（memory-match）**

| パラメータ | 記号 | 範囲 | 単位 |
|-----------|------|------|------|
| ペア数 | `pairs` | 2-24 | 組 |
| カード視覚類似度 | `similarity` | 0-100 | % |
| 裏返し猶予時間 | `flip_delay` | 800-2000 | ms |
| カードサイズ | `card_size` | 64-96 | px |

**分類ソーティング（sorting）**

| パラメータ | 記号 | 範囲 | 単位 |
|-----------|------|------|------|
| カテゴリ数 | `categories` | 2-5 | 個 |
| アイテム数 | `items` | 3-15 | 個 |
| 基準種類 | `criterion` | color/shape/size/category/multi | enum |
| 基準切替 | `switching` | none/between_rounds/within_round | enum |

**視覚探索（visual-search）**

| パラメータ | 記号 | 範囲 | 単位 |
|-----------|------|------|------|

| 配置 | `layout` | 不規則（固定） | - |

### 2.5 適応型難度調整アルゴリズム

```
【監視データ】
  直近10試行の正答率（一次判定）
  直近3セッションの平均正答率（トレンド判定）
  各試行のエラーレス学習到達Stage（補助指標）

【パラメータ調整ロジック】
  正答率 > 85% × 2セッション連続 → 主パラメータを1段階上げる
  正答率 < 65% × 2セッション連続 → 主パラメータを1段階下げる（サイレント）
  正答率 65-85% → 現状維持

  Stage 3ヒント到達率 > 30%（10試行中3回以上）
    → 正答率に関わらず、副パラメータを緩和（例: flip_delayを200ms延長）

【各ゲームの主パラメータ / 副パラメータ】
  神経衰弱: 主=pairs, 副=flip_delay, similarity
  分類:     主=categories/items, 副=criterion種類
  視覚探索: 主=diff_count/scene_items, 副=diff_subtlety
  Corsi:    主=seq_length, 副=display_ms, blocks

【自然な難度揺れ】
  同じパラメータ設定内でも、試行ごとに±10%のランダム変動を入れる
  → 子どもに「今回はちょっと違った」感を自然に与える
  → ただしパラメータ境界は超えない

【開始難度の推定】
  初回プレイ: Band B2（IQ 44-50相当）から開始
  最初の5-10試行の正答率で自動調整:
    全問正解 → 2バンド上へ移動
    70-90% → 1バンド上へ
    50-70% → 現バンド維持
    30-50% → 1バンド下へ
    30%以下 → 2バンド下へ
```

### 2.6 分類ゲーム: 基準切替の導入条件

基準切替はIQバンドではなく、**達成条件**で制御する。

```
【前提条件（全て満たすこと）】
  1. 単一基準・3カテゴリの分類で正答率85%以上を3セッション連続
  2. エラーレス Stage 3 到達率が10%以下（ほぼ自力で正解している）

【導入順序】

Step 1: ラウンド間切替
  1ラウンド目を完了 → 切替演出 → 2ラウンド目は別基準
  ・1ラウンド目: 色で分類
  ・切替演出（自動再生、3-5秒のアニメーション）:
    箱のアイコンが色→形に変化するアニメーション
    ナビキャラが新しいルールを指さし
  ・演出完了後、新しいアイテムが出現。子どもがタップして再開（時間制限なし）
  ・2ラウンド目: 形で分類

Step 2: ラウンド内切替
  条件: Step 1で正答率80%以上 × 3セッション連続
  ・1ラウンドの途中で基準が変わる
  ・切替の視覚的予告は必ず入れる（予告なし変更は禁止）
  ・子どものペースで再開

Step 3: 頻度増加
  条件: Step 2で正答率80%以上 × 3セッション連続
  ・1セッション内に2-3回の切替
```

### 2.8 保護者・教師向け表示（外部メトリクス）

**表示しないもの**: レベル番号、IQバンド、IQ値、他の子どもとの比較

#### 保護者向け（P2/P3画面）

各認知領域の**具体的到達度**を平易な言葉で表示:

```
おぼえて（ワーキングメモリ）
  「6組のペアを安定して見つけられるようになりました」
  先週: 4組 → 今週: 6組  ↑伸びています

なかまわけ（認知的柔軟性）
  「3つのグループに分けられます。新しいルールへの切り替えにも挑戦中です」

みつけて（注意力）
  「8つのものの中から2つの違いを見つけられます」

まねして（空間記憶）
  「5つのブロックの順番を覚えられるようになりました」
```

変化の方向（伸びている/安定している/ゆっくり進んでいる）のみ表示。下がっている場合は「今はこの内容を繰り返し練習中です」。

#### 教師・施設スタッフ向け（専門画面、将来実装）

| 表示項目 | 内容 |
|---------|------|
| 各パラメータの現在値 | pairs=6, similarity=40%, flip_delay=1200ms 等 |
| セッション別正答率推移 | 折れ線グラフ |
| 精神年齢換算（参考値） | 「空間WMは5歳8ヶ月相当」等 |
| エラーパターン分析 | 「色の弁別は得意だが、形の弁別に課題あり」等 |
| 難度パラメータ手動調整 | 上限・下限の設定。特定バンド内に固定も可能 |

## 3. ゲーム1: 神経衰弱（memory-match）

### 3.1 概要

| 項目 | 内容 |
|------|------|
| 認知機能 | ワーキングメモリ |


## 6. ゲーム4: Corsi Block（corsi-block）

### 6.1 概要

| 項目 | 内容 |
|------|------|
| 認知機能 | 空間ワーキングメモリ |
| 建物 | 光るグリッド塔 |
| 親向け表示名（仮） | まねして |
| 操作 | 観察フェーズ（見るだけ）→ 再現フェーズ（タップ） |
| 目標 | 光ったブロックの順番を正しく再現する |

### 6.2 操作方式

```
【観察フェーズ】
1. 不規則配置のブロックが1つずつ光る（1ブロック = display_ms時間点灯 → 消灯）
2. 全シーケンス完了後、0.5秒の間
3. ナビキャラが「やってみて」のジェスチャー

【再現フェーズ】
4. 子どもがブロックを順番にタップ
5a. 正しい順番 → ブロックが光る（観察時と同じ色）+ 正解フィードバック
5b. 間違った順番 → エラーレス学習発動

【完了】
6. 全シーケンス正解 → クリア演出
```

**フェーズ切替の視覚的区別**:
- 観察フェーズ: ブロックの枠線が白。タップしても反応しない
- 再現フェーズ: ブロックの枠線が黄色に変わる。タップ受付開始

```
│                    ○          ○                  │
│                                                  │
│              ○          ○                        │  ← 不規則配置
│                                   ○              │
│         ○                                        │
│                       ○                          │
│                                                  │
│      進捗: ●●○○ (シーケンス)       [ナビキャラ]  │

```

### 6.4 ブロック点灯のルール

```
色:
  デフォルト: くすんだグレー (#B0B0B0)
  点灯: 柔らかいゴールド (#C4A35A)
  正解タップ時: 点灯と同じゴールド

シーケンス生成ルール:
  Band A-B: 隣接ブロックのみで構成（空間的に追いやすくする）
  Band C-D: 隣接 + 1つ飛ばしまで許容
  Band E-F: ランダム配置（対角線移動含む）

```

```
           → シーケンスを最初からやり直し（ペナルティなし）

同じシーケンスで2回失敗:
  Stage 2: 観察フェーズを再表示（もう一度光の順番を見せる）
           → 再現フェーズ再開

同じシーケンスで3回失敗:
  Stage 3: 再現フェーズで、次にタップすべきブロックが柔らかく脈動
           → 子どもがタップ → 次のブロックが脈動 → 全シーケンス完了
           → 正解フィードバック（ヒント付きでもクリア扱い）
```

### 6.6 1試行の定義

**1試行 = 1シーケンスの正解再現**

## 8. データ記録項目（4ゲーム共通）

Phase 0-D-3（データ設計）の入力情報として。NCI算出の基盤データ（`nolla_nci_algorithm_design.md` 参照）。

### 8.1 試行データ（各試行で記録）

| データ | 型 | 用途 |
|--------|-----|------|
| game_id | enum | ゲーム種別 |
| params_snapshot | json | その試行時の全パラメータ値 |
| trial_number | int | セッション内の試行番号 |
| is_correct | boolean | 正解/不正解 |
| reaction_time_ms | int | タップまでの反応時間 |
| error_count | int | その試行内の不正解回数 |
| hint_stage_reached | int (0-3) | エラーレス学習のどの段階まで進んだか |
| session_duration_ms | int | セッション全体の所要時間 |
| timestamp | datetime | 試行の日時 |

### 8.2 タッチダイナミクスデータ（各タップで記録）【MVP】

タッチスクリーンAPIから追加コストゼロで取得可能。NCI精度向上とデジタルバイオマーカー研究の基盤。

| データ | 型 | 用途 |
|--------|-----|------|
| touch_force | float | タップ圧力。ASD児は有意に高い傾向（覚醒レベル・不安の推定） |
| touch_radius | float | 接触面積。指の使い方 → 運動発達の指標 |
| touch_duration_ms | int | 接触時間。即タップ vs 長押し → 確信度の推定 |
| touch_trajectory | json | タップ前の指の移動軌跡。迷い行動 → 認知負荷の直接測定 |
| inter_tap_interval_ms | int | 前のタップからの間隔。思考時間の精密測定 |
| tap_x, tap_y | float | タップ座標（正解位置からのズレ → 運動精度） |

**取得方法**: iOS の `UITouch` / Android の `MotionEvent` から全て取得可能。
**保存**: 生データで永久保存。分析は後からいくらでもできる。
**プライバシー**: タッチデータは個人識別リスクが低い。追加同意は不要（既存の行動データ同意でカバー）。

### 8.3 デバイスモーションデータ（セッション中常時記録）【MVP or Phase 1】

ジャイロスコープ・加速度センサーから取得。デバイスの動き = 子どもの身体状態の間接指標。

| データ | 型 | サンプリング | 用途 |
|--------|-----|-------------|------|
| gyro_x, gyro_y, gyro_z | float | 10Hz | デバイスの傾き変化 → 注意集中度の推定 |
| accel_x, accel_y, accel_z | float | 10Hz | 微振動 → 覚醒レベル、常同行動の検出 |
| jerk_magnitude | float | 算出値 | 加速度の変化率 → 急な動き（フラストレーション等）の検出 |

**取得方法**: iOS の `CMMotionManager` / Android の `SensorManager`。バックグラウンドで常時計測。
**サンプリングレート**: 10Hz（100ms間隔）で十分。バッテリー消費は無視できるレベル。
**保存**: セッション単位で集約統計値（平均、分散、ピーク）+ 生データ。

**NCI への統合**:
- デバイス安定度 → セッション品質指標（安定 = 集中、不安定 = 注意散漫）
- 常同行動パターン → 行動データとして保護者レポートに活用
- 異常セッション検出の精度向上（正答率 + 身体状態の複合判定）

### 8.4 環境データ（セッション開始時に記録）【Phase 2】

| データ | 型 | 用途 |
|--------|-----|------|
| ambient_noise_db | float | 環境騒音レベル → セッション品質補正（うるさい環境のスコアを割り引く） |
| time_of_day | enum | 朝/昼/夜 → 時間帯効果の推定 |
| days_since_last_session | int | 前回セッションからの経過日数 → プレイ頻度の影響分析 |

**マイクロフォン使用方針**:
- 環境騒音レベル（dB値のみ）: 録音なし、数値のみ取得。プライバシーリスク低
- 発声分析（Phase 2以降）: 要追加同意。発語のある子の言語発達モニタリング

### 8.5 ゲーム固有データ

| ゲーム | 追加データ | 用途 |
|--------|----------|------|
| memory-match | matched_pairs, total_flips, flip_positions | 記憶パターン分析 |
| sorting | category_errors (どのカテゴリ間で間違えたか) | 分類能力の弱点特定 |
| visual-search | tap_positions, distance_to_target | 視覚探索パターン |
| corsi-block | sequence_reproduced, error_position | 空間記憶の限界点��定 |

### 8.6 データ保存ポリシー

```
保存期間: 全データ Day1 から無期限保存（アカウント削除を除く）
理由: 取らなかったデータは後から取り直せない。
      取っておいたデータは後からいくらでも分析できる。

将来の活用:
  - 年次サマリレポート自動生成
  - データポータビリティ（他の療育機関への引き継ぎ）
  - 研究機関向け匿名化データAPI（Phase 3）
```
