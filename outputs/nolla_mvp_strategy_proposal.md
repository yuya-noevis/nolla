# Nolla MVP戦略提案

> 作成日: 2026-03-30
> 前提: ToC Webアプリ / 対象3-8歳 / Claude Codeで開発 / ToB展開は必須

---

## 1. ToC機能として最も需要が高いものTOP5

サービス数の多さ≠需要の高さ。以下は「親が日常的に直面する課題の深刻度 × 既存ソリューションの不十分さ × 3-8歳への適合度」を掛け合わせた評価。

### 1位: 親向けAIコーチング（行動対応・介入ガイド）

**なぜ1位か**: 17サービス中、親を「介入者として育てる」ことに本気で取り組んでいるのはAutism360とUnderstanding Zoeのみ。しかも両者ともLLMチャットボット止まり。一方で最新のエビデンス（2025年ネットワークメタ分析）は「親媒介介入（NDBI）が最も効果的」と明確に示している。つまり**需要は巨大だがまともな供給がない**。

- 困りごと: 癇癪・こだわり・切り替え困難が起きた「その瞬間」に、何をすればいいかわからない
- 既存の限界: チャットボットに状況を文字で打っている余裕はない。動画教材は汎用的すぎて「うちの子」に合わない
- AIで解決すべきこと: マルチモーダル（音声+映像）でリアルタイムに状況を理解し、その場で具体的な対応を指示する

### 2位: コミュニケーション支援（AAC + 発語促進）

**なぜ2位か**: 10+サービスが参入しているが、それは需要が大きいから。3-8歳の発達障害児で最も多い主訴が「ことばの遅れ」。ただし既存AACアプリ（Proloquo2Go等）は操作が複雑で、LLMの文脈理解を活用しているものがゼロ。発語なし〜少語の子どもに対して、AIが「次に言いたいこと」を予測する精度は今の技術で劇的に改善できる。

- 困りごと: 子どもが何を伝えたいのかわからない。絵カードでは表現の幅が狭すぎる
- 既存の限界: 静的な絵カードの羅列。文脈を理解しない。親が手動でカスタマイズし続ける必要がある
- AIで解決すべきこと: 場面・時間・履歴から「今この子が伝えたいこと」を予測し、最小タップで表現できるようにする

### 3位: 日常ルーティン・見通し支援（ビジュアルスケジュール）

**なぜ3位か**: Goally、Tiimo（App Store年間最優秀アプリ）が証明するように、「見通しが立たない不安」は障害種別を問わず最も普遍的な困りごと。しかし既存サービスは親が手動でスケジュールを作る必要があり、それ自体が負担。生成AIでの自動化は技術的に即座に実現可能で、効果も高い。

- 困りごと: 予定変更でパニック。朝の支度が毎日戦争。「次に何をするか」がわからないと動けない
- 既存の限界: 親がスケジュールを毎日手作りする負担。子どもの理解レベルに合った視覚表現を親が選ぶ必要がある
- AIで解決すべきこと: カレンダー連携+生活パターン学習で自動生成。予定変更を検知して事前にソーシャルストーリーで心理的準備を促す

### 4位: 感情調整・メルトダウン予防

**なぜ4位か**: 7+サービスが参入しているが、「予防」に成功しているものがない。すべて「起きた後の対処」止まり。最新のウェアラブル+AI研究（KeepCalm等）は、生体データからメルトダウンを事前に予測できる可能性を示している。予防できれば親の精神的負担は劇的に減る。

- 困りごと: いつメルトダウンが起きるか予測できず、常に緊張状態。外出が怖い
- 既存の限界: 事後対応のみ。感情認識アプリはあるが「だから何をすればいいか」まで到達しない
- AIで解決すべきこと: 行動パターン+環境データ+履歴から「あと10分でしんどくなりそう」を予測し、事前にクールダウン戦略を親に提案

### 5位: 発達の可視化・進捗トラッキング

**なぜ5位か**: 親の最大の不安は「うちの子、成長しているのか？」。療育に通っても具体的な進捗が見えにくい。既存サービスは「ゲームのスコア」レベルのトラッキングしかしておらず、発達全体を俯瞰できるものがない。日本の児童発達支援では半年ごとのモニタリングが義務だが、その間のデータは空白。

- 困りごと: 療育の効果がわからない。先生に「成長してますよ」と言われても具体的に何が？
- 既存の限界: 活動スコアの記録止まり。発達領域を横断した総合的なトラッキングができない
- AIで解決すべきこと: 日常の使用データから、発達領域（言語・社会性・認知・運動・日常生活）ごとの変化を自動検出し、親にも専門家にもわかる形で可視化

## 2. MVP実装優先順位TOP5

前提: 既存サービスはAIのポテンシャルを引き出せていない。AIエージェントとしての機能を持たないサービスは淘汰される。ここでは「最新AI技術をフル活用し、かつMVPとして最小限の実装で最大の価値検証ができるもの」を優先。

### P1: AIペアレントコーチ（親向けエージェント）

**これがMVPの核。他のすべてはこの上に乗る。**

| 項目 | 内容 |
|------|------|
| 何をするか | 子どもの特性プロファイルを持ったAIエージェントが、親の「今この瞬間の困りごと」に対して、NDBI原則に基づいた具体的な対応を提案する |
| AI技術 | LLMエージェント（長期記憶+子どものプロファイル+エビデンスDB）。音声入力対応（片手が塞がっていても使える） |
| なぜP1か | **親が介入者として機能しなければ、子ども向け機能はすべて無意味**。3-8歳の子どもは一人でアプリを適切に使えない。親のスキルアップが最も費用対効果が高い（NDBI研究で実証済み） |
| MVP検証 | 10家庭に使ってもらい、「実際の困り場面で役に立ったか」を定性評価 |
| 技術的実現性 | Claude API + RAG（エビデンスDB）+ 音声入力（Whisper）で構築可能。Webアプリで即座にデプロイ可能 |

**既存サービスとの決定的な違い**:
- Understanding Zoe: テキストチャットボット → Nolla: 音声入力+マルチモーダル対応のエージェント
- Autism360: 汎用的なアドバイス → Nolla: 「この子」のプロファイルに完全に個別化された対応指示
- どちらもエージェントではない（プロアクティブに提案しない、長期記憶がない、行動計画を立てない）

### P2: 子どもプロファイルエンジン（認知・行動・感覚の個別化モデル）

| 項目 | 内容 |
|------|------|
| 何をするか | 子ども一人ひとりの「認知プロファイル」を構築・更新し続けるエンジン。診断名ではなく、具体的な強み・困難・感覚特性・興味を構造化データとして保持 |
| AI技術 | 親からの入力（初期質問+日常記録）をLLMが構造化。使用データからの自動学習。ベイズ推定でプロファイルを継続更新 |
| なぜP2か | P1（親コーチ）もP3以降もすべて、「この子を知っている」ことが前提。汎用的なアドバイスでは既存サービスと差別化できない。プロファイルの精度がNollaの全機能の質を決定する |
| MVP検証 | プロファイルに基づく提案と、汎用的な提案のA/Bテストで精度を検証 |

**これが全機能の「背骨」になる。ToB展開時にも、このプロファイルデータが施設・学校との連携の基盤になる。**

### P3: AIビジュアルスケジュール自動生成

| 項目 | 内容 |
|------|------|
| 何をするか | 子どものプロファイル（理解レベル・感覚特性・興味）に基づき、毎日のスケジュールをAIが自動生成。予定変更時は事前通知+ソーシャルストーリーを自動作成 |
| AI技術 | 生成AI（テキスト→画像→スケジュールUI）。カレンダーAPI連携。プロファイルエンジン（P2）からの個別化 |
| なぜP3か | 実装コストが低く、効果が即座に実感できる。毎日使う機能なので継続率に直結する。Tiimoが2025年App Store年間最優秀を取ったことが需要を証明。しかしTiimoはAIによる個別化が弱い |
| MVP検証 | 「手動で作るより楽か」「子どもの理解度に合っているか」を検証 |

### P4: 発達トラッキングダッシュボード

| 項目 | 内容 |
|------|------|
| 何をするか | P1〜P3の使用データ+親の日常記録から、発達領域ごとの変化を自動検出・可視化。月次レポートを自動生成 |
| AI技術 | 時系列分析+NLP（親の記録テキストからの情報抽出）+生成AI（レポート自動生成）。プロファイルエンジン（P2）と連動 |
| なぜP4か | 「効果が見える」ことが継続利用の最大の動機。また、このレポートがToB展開の入口になる（施設に共有→施設が価値を認識→施設契約） |
| MVP検証 | 「このレポートは療育の先生に見せたいか」を検証 |

### P5: AIコミュニケーションボード（次世代AAC）

| 項目 | 内容 |
|------|------|
| 何をするか | LLMの文脈理解を活用した、発語なし〜少語の子どものためのコミュニケーション支援。場面・時間・履歴から「今伝えたいこと」をAIが予測し、最小操作で表現できるUI |
| AI技術 | LLM（文脈予測）+プロファイルエンジン（P2）+画像生成AI（個別化シンボル）。将来的にはカメラ入力で視線操作も |
| なぜP5か | 発語のない子どもにとってはコミュニケーション手段そのもの。ただしMVPでは対象ユーザーが限定されるため優先度は5位。P1-P4が広いユーザー層をカバーした上で、特化機能として追加 |
| MVP検証 | 既存AAC（Proloquo2Go等）との比較で「予測精度」と「操作ステップ数」を検証 |

## 3. ビジネスモデル: ToC → ToB の導線設計

### ToC（家庭向け）

| 要素 | 内容 |
|------|------|
| 料金 | フリーミアム。基本機能（スケジュール・簡易プロファイル）は無料。AIコーチング・詳細トラッキングは月額課金 |
| 価格帯 | 月額¥1,980〜¥3,980（海外競合: $10-30/月。日本の療育費自己負担がほぼゼロなので、価格感度が異なる点に注意） |
| 獲得チャネル | SNS（当事者コミュニティ）→ 口コミ。療育施設の待合室。小児科・発達外来の紹介 |

### ToB（施設・学校向け）

**ToC→ToBの導線が鍵。親が「これ施設にも見せたい」と思うことがトリガー。**

| 要素 | 内容 |
|------|------|
| 対象 | 児童発達支援事業所・放課後等デイサービス（全国約4万事業所） |
| 提供価値 | ①子どもの家庭データが施設に共有される（個別支援計画の質向上）②発達レポートの自動生成（記録業務の削減）③エビデンスベースの介入提案（支援の属人性排除） |
| 料金 | 児童1人あたり月額 or 事業所単位の月額SaaS |
| 導線 | 家庭で使っている親が施設に「Nollaのレポート見てください」と共有 → 施設が価値を認識 → 施設契約 → 施設から他の家庭にも推薦（フライホイール） |

### ToB拡張: 学校（ToG経由）

| 要素 | 内容 |
|------|------|
| 対象 | 特別支援学校・通級指導教室（自治体経由） |
| 提供価値 | IEP（個別の教育支援計画）の自動ドラフト生成。家庭と学校のデータ連携 |
| 時期 | ToC・施設ToBが確立した後。自治体営業には実績が必要 |

## 4. Nollaの理想的なサービス全体像

### コンセプト

**「この子を最もよく知っているAI」が、家庭・施設・学校を横断して発達を支える統合プラットフォーム**

既存サービスが「単機能ツール」の集合体にとどまっている中、Nollaは子どもの認知プロファイルを中核に据え、すべての機能がそのプロファイルから個別化される設計を取る。

### アーキテクチャ

```
                        ┌──────────────────────┐
                        │   子どもプロファイル    │
                        │   エンジン (P2)        │
                        │                      │
                        │  認知・行動・感覚・    │
                        │  興味・発達履歴        │
                        └──────────┬───────────┘
                                   │
                 ┌─────────────────┼─────────────────┐
                 │                 │                 │
        ┌────────▼───────┐ ┌──────▼──────┐ ┌────────▼────────┐
        │  親向けレイヤー  │ │ 子向けレイヤー│ │ 専門家レイヤー    │
        │                │ │             │ │                 │
        │ AIコーチ (P1)   │ │ スケジュール │ │ 発達レポート     │
        │ 発達ダッシュ(P4)│ │ (P3)        │ │ IEP生成         │
        │ コミュニティ    │ │ AAC (P5)    │ │ 介入効果分析     │
        │                │ │ 学習活動     │ │ 記録・請求連携   │
        └────────────────┘ │ ソーシャル   │ └─────────────────┘
                           │ ストーリー   │
                           └─────────────┘
```

### 段階的ロードマップ

| フェーズ | 内容 | 目標 |
|---------|------|------|
| **Phase 1: MVP** | P1（AIコーチ）+ P2（プロファイル）+ P3（スケジュール） | 10家庭で検証。「役に立った」の定性評価 |
| **Phase 2: 検証** | P4（発達トラッキング）追加。ToC課金開始 | 100家庭。継続率・課金転換率の検証 |
| **Phase 3: ToB** | 施設向け連携機能。レポート共有。記録業務削減 | 5施設との契約。親→施設の導線検証 |
| **Phase 4: 拡張** | P5（AAC）。音声対話。IoT連携。学校向け | 1,000家庭 + 50施設。シリーズA準備 |

### 競争優位性（なぜNollaが勝てるか）

1. **プロファイル中心設計**: 機能の寄せ集めではなく、「この子」のデータが全機能を個別化する。使えば使うほど精度が上がり、乗り換えコストが高くなる
2. **エージェントアーキテクチャ**: チャットボットではなく、プロアクティブに提案・行動するAIエージェント。長期記憶を持ち、「この子」の変化を追い続ける
3. **家庭↔施設のデータブリッジ**: 家庭と施設のデータが断絶している現状を解決する唯一のプラットフォーム。日本の福祉制度（個別支援計画・モニタリング）との統合はローカル参入障壁になる
4. **エビデンスの組み込み**: NDBI原則をAIの行動指針に直接組み込む。「なんとなくAI」ではなく「科学的に正しいAI」
5. **日本市場の先行者優位**: ToC向けAI発達支援アプリは国内にほぼ存在しない。チャッピーくん（開発中）、CASP（プロトタイプ）のみ

## 補足: 議論の中で出た重要な論点

### CPO視点: 「親向けが先、子ども向けは後」の理由
3-8歳の子どもは自力でアプリを操作しない。親がセットアップし、一緒に使うか、結果を見る。つまりUXの主体は親。子ども向け機能を先に作っても、親が使い方を理解していなければ機能しない。Understanding Zoeが親向けに絞って$770K調達した判断は正しい。

### COO視点: MVP最小構成の定義
P1+P2+P3の3機能でMVP。P1（AIコーチ）が単体で価値を提供でき、P2（プロファイル）がその精度を上げ、P3（スケジュール）が毎日の接点を作る。この3つが揃えば「毎日使う理由」と「使い続ける理由」が成立する。

### CMO視点: 日本市場特有の注意点
日本の療育は自治体負担で自己負担がほぼゼロ。「無料で受けられる療育」と比較される。Nollaは「療育の代替」ではなく「療育では届かない週166時間を埋める」というポジショニングが必須。「療育に通っているのに、家では何もできていない」という罪悪感がフックになる。

### Clinical Advisor視点: エビデンスの重要性
Nooraが差別化できているのはRCTがあるから。MVPの段階からデータ収集設計を組み込み、将来のエビデンス構築に備えるべき。ただし、AIが生成する介入提案には必ずエビデンスレベルを明示し、「確立された手法」と「可能性のある手法」を区別すること。LLMのハルシネーションで誤った療育アドバイスを出すリスクは最大の注意点。
