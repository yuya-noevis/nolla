# Nolla 自己検証チェックリスト (修正→デプロイ前に必ず通す)

> 修正のたびに Claude 側でここを**必ず全項目通過**してから Yuya に報告する。
> 未通過のままYuya にテスト依頼することは禁止。
> 該当しない項目は理由付きで skip と記録する。

## A. 自動検証 (修正ごとに毎回)

- [ ] A1. `vitest run` → 全テスト pass (296+/296+)
- [ ] A2. `tsc --noEmit` → 修正/新規ファイルにエラーなし
- [ ] A3. `next build` → exit 0
- [ ] A4. 新規/変更ロジックにユニットテストを追加した (既存バグの再発防止)

## B. コードレビュー (修正ごとに毎回)

- [ ] B1. 変更した `useEffect` の依存配列を1つずつ確認: 参照等価の不安定な値 (配列/オブジェクト/毎レンダ生成関数) が入っていないか
- [ ] B2. 変更した React state の更新で、setState updater 内から外側変数へ値を書き込んでいないか (BUG-15 再発防止)
- [ ] B3. `async`/`Promise` の race condition: ref 更新タイミングと read タイミングがズレていないか
- [ ] B4. 副作用 (DB write / localStorage / router.push) が条件分岐ミスで呼ばれ過ぎ/呼ばれな過ぎしていないか
- [ ] B5. `| null` `undefined` のガードが修正箇所に入っているか

## C. UX/仕様 (UI変更時)

- [ ] C1. 各画面で「ホームに戻る」動線が壊れていないか (ボタン存在・動作)
- [ ] C2. カルーセル/タブ/スライダーのインタラクション: 左右・前後・戻るが全て動くか (片方しか動かない等の半壊状態禁止)
- [ ] C3. タップターゲット 48px 以上
- [ ] C4. 初期状態の表示 (初回ユーザーと再訪ユーザーで想定通りか)
- [ ] C5. スター/ポイント計算が「ユーザーが期待する結果」と一致するか (仕様通りでもUX破綻していないか要確認)
- [ ] C6. 閾値・時間制限・難度が対象年齢と知的レベルに合っているか

## D. ゲーム共通 (ゲームロジック変更時)

- [ ] D6. スター付与が `stars` テーブルに記録される

## E. 既存機能の回帰確認 (自動化したいが現状は手動)

## F. デプロイ前の最終確認

- [ ] F1. Supabase の過去データで「意味のある」テストになっているか (必要なら stale row を削除)

## 運用ルール

1. 修正作業の最初にこのファイルを開き、該当カテゴリをリストアップ
2. 修正完了後、該当項目を1つずつ通過
3. 通過できない項目は「なぜ通過できないか」を記録し、修正を継続
4. 全通過したら commit → push → Yuya に報告
5. 未通過の項目がある状態で Yuya にテスト依頼したら開発リードとして失格
